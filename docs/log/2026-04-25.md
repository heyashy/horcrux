- [CHORE] Phase 2.6 — Character-discovery hardening (multiple iterations
  V8-V15 in the same session, captured as Findings 9-15):
  - **Memory streaming** in `resolve_coref_aliases`: process chapters
    one at a time with periodic `gc.collect()` + `torch.cuda.empty_cache()`,
    bounding peak memory to single-chapter inference state. Required for
    full-corpus runs (200+ chapters of ~30KB each OOM'd batched call).
  - **Stop-modifier filter** (`_STOP_MODIFIERS` + extended
    `_is_alias_candidate`): rejects `Voldemort himself`, `Dear Harry`,
    `only Crookshanks`, `poor old Ripper`, `Harry blankly`, etc. Applied
    BOTH at Tier 1b coref-attribution AND at Tier 1a `count_mentions`
    so prose-fragment NER false positives never form their own clusters.
  - **Min-anchor-count rule** in `cluster_aliases`: multi-word names
    with 3+ significant tokens require count ≥ 5 to bridge subset
    merges. Prevents rare full-name forms (Harry James Potter, count=2)
    from transitively merging two distinct two-word names (Harry Potter
    + James Potter).
  - **Tier 1c rule consistency**: `claim_single_word_clusters` excludes
    3+-significant-token canonicals from its owner index. Same anchor
    concept applied at every stage that uses it.
  - **Corpus-wide person_text_set** in `resolve_coref_aliases`:
    pre-pass NER over all chapters, accumulate `corpus_person_texts`,
    use it in attribution. Prevents same-surname siblings from leaking
    attributions across chapters where one is NER-tagged and the other
    isn't (Albus Dumbledore was being merged into the Ariana Dumbledore
    cluster). Cached per-chapter NER spans to avoid redundant work.
  - **Rolled back** an over-fit Levenshtein/first-name-conflict rule —
    same orthographic feature shape applies to "different siblings"
    (Albus/Ariana, block) AND "nickname vs given name" (Mad-Eye/Alastor,
    allow). No purely orthographic rule discriminates correctly.
    Captured as Finding 13.
  - **Dropped Tier 2 LLM merge** entirely. Non-determinism at ingest
    time has too large a blast radius (every chunk's character payload
    drifts across runs). Tier 3 manual overrides handle semantic
    aliases instead. Captured as Finding 11.

- [CHORE] Phase 3 design notes:
  - Two collections (hp_chapters, hp_paragraphs) instead of one
    mixed collection + chunk_type filter. Keeps the chapter working
    set small (~200 points) for breadth-style queries; lets the two
    be tuned independently if it ever matters.
  - Embedding runs raw (not via Temporal). Embedding's failure profile
    doesn't earn Temporal: deterministic, idempotent UUID5 IDs, single
    process, single model load. OCR earned Temporal because it's long,
    parallel, and partial-progress recovery matters; embedding doesn't.

- [CHORE] `docs/lab/findings.md` — two new findings:
  - F17: Embedding models silently truncate at the context window.
    bge-large's 512-token cap means "chapter-level" embeddings are
    really "chapter-opening-level" embeddings. Three layers of
    abstraction (embedder API / vector store / app code) pass
    truncation through silently. Lab accepts the limitation; pool-
    over-windows is the standard fix if it bites in evaluation.
  - F18: Qdrant silently brute-forces below
    `optimizer_config.indexing_threshold` (default 20,000). Both
    collections are below threshold so HNSW config (m=16,
    ef_construct=100) is declarative-only; queries do exhaustive
    cosine. Brute-force is genuinely faster + more accurate at this
    scale — but the gap between "config says HNSW" and "server
    actually does HNSW" is a real failure mode for anyone
    benchmarking ANN tuning on a small dataset.
