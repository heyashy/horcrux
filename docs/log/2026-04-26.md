# Change Log — 2026-04-26

## Summary
Phase 4.7 — query rewriter + conversational CLI. Closes the typo case
from end-of-day testing yesterday (Finding 22 candidate) at the input
layer, and adds multi-turn chat with PydanticAI message-history threading.

## Changes
- [CHORE] Phase 4.7 — Query rewriter + conversational CLI. Files added:
  - `horcrux/query_rewrite.py` — fuzzy keyword correction over the
    corpus vocabulary. Pure deterministic function, no LLM call,
    ~milliseconds per query. Vocab built once from `chunks.json` via
    `lru_cache`, ~50k unique tokens for the HP corpus. Threshold-tuned
    rapidfuzz matching: out-of-vocab tokens >=5 chars get fuzzy-matched
    against vocab; matches scoring >=80 are substituted, anything below
    passes through. Closes Finding 22 candidate (typo robustness).
  - `horcrux/conversation.py` — `ChatSession` and `Turn` dataclasses.
    Pure data structures, no I/O. UUID thread_id per session, append-
    only history list. `record()` returns the appended Turn for
    chained rendering. `reset()` clears history but keeps thread_id
    (intentional — same logical conversation, just no carryover).
  - `scripts/chat.py` — REPL with Rich. Per-turn flow: prompt →
    retrieve → synthesise (with message_history) → render → loop.
    Slash commands: `/exit`, `/quit`, `/q`, `/clear`, `/history`,
    `/help`. Unknown commands flagged. Catch-all error handler so a
    rate-limit or malformed-output blip doesn't kill the REPL.
  - `tests/unit/test_query_rewrite.py` — 12 tests: in-vocab passthrough,
    short-OOV no-correction, real typo correction, multiple typos,
    threshold behaviour at extremes, tokenisation edge cases.
    Tests pass an explicit small `vocab` so behaviour doesn't couple
    to the gold-tier `chunks.json`.
  - `tests/unit/test_conversation.py` — 10 tests: thread_id uniqueness,
    history append, record return value, reset behaviour (clears
    history, keeps thread_id), turn payload preservation.
- [CHORE] Phase 4.7 — files changed:
  - `horcrux/retrieval_graph.py` — added a `rewrite` node before the
    four retrievers; state extended with `original_query` and
    `corrections`. New `RetrievalResult` dataclass replaces the bare
    `list[ScoredCandidate]` return from `hybrid_search`, exposing the
    rewriter's view alongside the candidates. The dataclass implements
    `__iter__`, `__len__`, `__bool__` so existing iteration and
    truthiness checks keep working.
  - `horcrux/agents.py` — `synthesise()` kept as the one-shot API
    (unchanged signature for `make answer`). Added
    `synthesise_with_history(query, candidates, *, message_history)`
    that returns `(Finding, all_messages)` for chat use. Internally
    `synthesise()` delegates to the history variant with `None`.
  - `scripts/answer.py` + `scripts/search.py` — unpack
    `RetrievalResult.candidates` and surface `result.corrections` as a
    "did you mean…" note when the rewriter applied any.
  - `Makefile` — `make chat` target. Help text updated.
- [CHORE] Phase 4.7 verification (live, end-to-end against LiteLLM

- [CHORE] Phase 6 — Research mode (multi-step planning + parallel
  sub-queries + aggregator) with streamed reasoning visible in the
  CLI. Files added:
  - `horcrux/models.py`: `Plan` (1-8 sub-questions, schema-bounded),
    `SubFinding` (per-branch result), `ResearchReport` (full reasoning
    chain — plan + sub_findings + final answer/citations/conviction).
  - `horcrux/planner.py`: Haiku-driven decomposition agent. Cheap +
    fast (decomposition is structural not capability-bound). Cached
    per-process via `lru_cache`.
  - `horcrux/aggregator.py`: Sonnet-driven aggregator. Reads sub-findings
    + a flattened deduplicated candidate set, produces final coherent
    Finding with strict-RAG enforced at this layer too (numbered
    citations, runtime in-range check via `_resolve_citations`).
  - `horcrux/research_graph.py`: top-level LangGraph using `Send` for
    parallel fan-out. First place in the codebase that genuinely
    earns `Send` (vs the existing `asyncio.gather` pattern) — each
    sub-query needs to be independently visible in `astream` events.
    `Annotated[list, operator.add]` reducer on `sub_findings` lets
    parallel branches append without races.
  - `horcrux/research_renderer.py`: `StreamingRenderer` consumes
    `astream(stream_mode='debug')` events and prints node-lifecycle
    progress live; `render_report()` for static post-hoc rendering
    used by `/trace`.
  - `scripts/research.py` + `make research Q="..."`: standalone CLI.
- [CHORE] Phase 6 — files changed:
  - `horcrux/embedding.py`: double-checked-locking guard around the
    bge-large `SentenceTransformer` constructor. Caught in the first
    parallel research run when 3 sub-queries each tried to construct
    their own model and OOM'd the GPU. The lock ensures exactly one
    constructor runs even under parallel first-callers.
  - `scripts/chat.py`: `/research <question>` and `/trace` slash
    commands. REPL state factored into `_ReplState` dataclass +
    `_dispatch_command` helper to keep `_amain` clean.
  - `Makefile`: `make research` target.
  - `docs/adr/MANIFEST.md`: ADR-0009 added.
- [ADR-0009] Research mode — planner + parallel sub-queries +
  aggregator. Captures the Plan-Execute-Aggregate pattern, why `Send`
  earns its keep here vs `asyncio.gather` (per-branch visibility),
  the rationale for split planner=Haiku / aggregator=Sonnet (cheap
  decomposition vs capable synthesis), the four rejected alternatives
  (long-context, gather-only, recursive planning, single-model-for-
  all), and the empirical Snape-arc results.
- [CHORE] Phase 6 — Tests + verification:
  - `tests/unit/test_research.py`: 12 unit tests covering Plan/
    ResearchReport schema, candidate merging, planner agent (mocked
    via TestModel), aggregator runtime citation check.
  - 203 unit tests passing total (was 191; +12 for research).
- [CHORE] Phase 6 verification — live end-to-end against LiteLLM proxy
  + Qdrant on the motivating query "tell me about Snape's story arc":
  - Planner produced 4 well-chosen sub-questions covering early-book
    role / Prince's Tale / Dumbledore's killing / Harry's perspective
    shift. Rationale was coherent.
  - 4 sub-queries ran in parallel, ~30-36s each, 10 hits per, with
    properly distributed convictions (5/5 for direct quote material,
    4/5 for clear implication, 3/5 for partial coverage).
  - Aggregator returned conviction 4/5 — bounded by the weakest
    sub-finding, exactly the F20 calibration pattern.
  - 19 unique citations spanning all 7 books in the final answer
    (multi-book synthesis the single-shot path cannot do).
  - `gaps` field honestly listed three absences (Prince's Tale memory
    contents, Marvolo Gaunt ring detail, Harry's emotional aftermath).
  - The streamed UX showed every step: plan → 4 parallel "running…"
    spinners → completion lines with hits/cited/conviction/elapsed →
    "Synthesising final report…" → final answer panel with cited
    passages. The agent's reasoning is visible end-to-end.
- [CHORE] Architectural insight worth capturing: **F20 (conviction
  calibration anchoring high) is organically resolved by the planner-
  aggregator architecture** without prompt-tuning. Each sub-finding has
  its own conviction; the aggregator is bounded by the weakest. The
  Snape question came back at 4/5 because one sub-finding is 3/5 — the
  single-shot path on the same question would have returned 5/5.
